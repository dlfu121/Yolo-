构建一个最简化可行产品（MVP）的 PRD（产品需求文档），核心原则是**“砍掉一切不影响核心业务逻辑验证的边缘功能”**。针对你的视觉健身质量评估项目，MVP 的唯一目的是验证：**系统能否通过单摄像头准确识别动作、捕捉运动极值（角度与速度），并输出有价值的评估报告。**
以下为你量身定制的 MVP 阶段 PRD 框架：

---

## 核心业务需求文档 (MVP-PRD)：智能健身动作质量评估 Demo

### 1. 产品概述与目标

* **产品名称**：FitVision MVP (暂定)
* **产品目标**：跑通基于单目视觉的“动作识别 - 质量评估 - 报告生成”完整技术闭环。
* **核心价值**：不仅能计数（Rep-counting），更能通过关节角度和离心/向心运动速度分析，为用户提供动作完成质量的客观反馈。
  
  ### 2. 目标用户与使用场景
* **目标用户**：开发测试人员（内部验证）、少量种子健身用户。
* **使用场景**：用户在光线充足的室内，将单摄像头（如笔记本电脑前置或外接 WebCam）放置在正侧方（矢状面）或正前方（冠状面），在镜头视野内完成指定动作。
  
  ### 3. MVP 范围界定 (Scope)
  
  为保证快速落地，初版 Demo **不包含**以下内容：
* ❌ 复杂的用户登录与云端账号系统。
* ❌ 多摄像头三维重建协同。
* ❌ 移动端 App 打包部署（初版直接在本地 PC 终端或简单 Web 页面运行）。
* ❌ 涵盖所有健身动作（初版仅硬编码支持 1-2 个标准动作，如深蹲或二头肌弯举）。
  
  ### 4. 核心功能需求
  
  #### 4.1 视频输入与追踪模块 (Vision Input & Tracking)
* **视频流接入**：支持调取本地 USB 摄像头实时画面或读取本地 `.mp4` 测试视频。
* **姿态估计提取**：实时提取人体 17 个关键点坐标。在 MVP 阶段的开发与测试环境中，为保证 YOLOv8-Pose 模型在无压缩状态下实现极低延迟和高帧率推理，系统架构将直接针对高性能本地算力环境（如双 NVIDIA RTX 4090 GPU 节点）进行吞吐量和显存调用优化。
* **置信度过滤**：当关键点（如髋、膝、踝）检测置信度低于 0.5 时，系统暂停当前帧的角度计算，防止脏数据干扰。
  
  #### 4.2 动作评估逻辑引擎 (Quality Evaluation Engine)
* **状态机运转**：系统需内置目标动作的标准状态机（如：准备 -> 离心下放 -> 最低点 -> 向心推起）。
* **角度监测 (Angle Tracking)**：
  * 实时计算核心关节夹角（如深蹲时的膝关节）。
  * 设定达标阈值（例如：膝角需 $< 90^\circ$ 判定为深度合格）。
* **速度与节奏监测 (Tempo Tracking)**：
  * 记录动作各阶段的耗时（利用帧率与帧数换算）。
  * 设定时间阈值（例如：离心下放阶段耗时 $< 1.0$ 秒则判定为“缺乏控制/速度过快”）。
    
    #### 4.3 UI 交互与实时反馈 (UI & Real-time Feedback)
* **骨骼渲染**：在视频画面上实时绘制人体骨架。
* **数据看板 (HUD)**：画面边缘实时显示当前关节角度数值、当前动作阶段、已完成的有效次数。
* **视觉高亮**：当动作出错（如速度过快或角度不足）时，相关关节的骨骼线段闪烁红色以示警告。
  
  #### 4.4 运动报告生成 (Report Generation)
* 运动结束后，系统生成结构化报告（可先输出为 JSON 或终端打印的文本格式）。
* **报告核心字段**：
  * 总次数与有效次数。
  * 平均动作耗时（节奏表现）。
  * 最低点平均角度。
  * **改进建议**（核心亮点，如：“在第 3 次和第 4 次动作中，下放速度过快，请加强离心阶段的肌肉控制”）。

---

### 5. 系统架构与技术选型建议

| 模块       | 推荐技术栈          | 备注                               |
|:-------- |:-------------- |:-------------------------------- |
| **视觉底层** | YOLOv8-Pose    | 抗遮挡性强，关键点提取稳定。                   |
| **核心逻辑** | Python, NumPy  | 用于处理坐标系转换、向量计算和状态机。              |
| **画面渲染** | OpenCV (`cv2`) | 快速绘制边界框、文字和骨架连线。                 |
| **规则配置** | YAML 或 JSON    | 将“动作阈值（角度、时间）”与核心代码解耦，方便后期扩展新动作。 |

### 6. 验收标准 (Acceptance Criteria)

1. 输入一段标准的深蹲测试视频，系统能正确计数，误差率 $< 5\%$。
2. 输入一段“半蹲”或“下放速度极快”的错误动作视频，系统能够成功识别不达标，且不在有效次数中增加计数。
3. 程序结束运行后，能立即输出包含角度和节奏评价的汇总报告。
